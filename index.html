    • Requisitos Funcionales: Describen las funciones y operaciones que el sistema debe realizar.
    • Requisitos No Funcionales: Se refieren a criterios de rendimiento, seguridad, y otras características no directamente relacionadas con las funciones específicas del sistema.
  4. Técnicas de Recolección de Información: La obtención de información es esencial para entender los requisitos del sistema. Dos técnicas comunes son las entrevistas y la lluvia de ideas
    • Entrevistas: Se utilizan para obtener información directa de las partes interesadas, como usuarios y expertos en el dominio del problema.
    • Lluvia de ideas: Permite la generación libre de ideas para identificar posibles soluciones o requerimientos del sistema.
  5. Los Casos de Uso: Los casos de uso son una herramienta valiosa para describir las interacciones entre el sistema y sus actores.
    • Los Actores: Representan roles externos que interactúan con el sistema.
    • El Caso de Uso: Describe una secuencia de acciones que un sistema realiza para producir un resultado observable.
    • Las relaciones: Asociación, Inclución, Extensión, Generalización y Especialización
    • Documentación del Caso de Uso: La documentación detalla la información esencial sobre el caso de uso, proporcionando una referencia clara.

Ingeniería de Requisitos

La ingeniería de requisitos es un proceso sistemático y colaborativo que se centra en la identificación, análisis, documentación, verificación y gestión de los requisitos necesarios para el desarrollo de un sistema de software. Este proceso busca comprender y traducir las necesidades de los usuarios y partes interesadas en requisitos claros y verificables, estableciendo una base sólida para el diseño, la implementación y la evaluación exitosa de un sistema informático. La ingeniería de requisitos abarca desde la recopilación inicial de información hasta la gestión continua de los requisitos a lo largo del ciclo de vida del desarrollo, asegurando la alineación constante entre el software producido y las expectativas de los usuarios y las metas del proyecto.

  1. Elicitación de Requisitos : La elicitación busca descubrir, comprender y documentar las necesidades de los usuarios y partes interesadas.
    • Métodos: Entrevistas, donde se recopila información directa; observación de procesos existentes; encuestas para obtener opiniones diversas; y talleres de trabajo para fomentar la colaboración.
  2. Análisis de Requisitos: En esta etapa, se examinan los requisitos recopilados para identificar posibles problemas como inconsistencias, ambigüedades o requisitos conflictivos.
    • Objetivo: Clarificar y refinar los requisitos, garantizando su comprensión antes de pasar a la fase de especificación.
  3. Especificación de Requisitos: La especificación implica documentar los requisitos de manera clara y completa para que puedan ser entendidos y aplicados.
    • Documentos: Pueden utilizarse diferentes artefactos, como casos de uso, diagramas UML, historias de usuario o especificaciones formales, según la naturaleza del proyecto.
  4. Verificación de Requisitos: La verificación asegura que los requisitos sean correctos, completos y coherentes.
    • Herramientas: Revisión por pares, validación cruzada y pruebas de concepto son técnicas comunes. Las herramientas de gestión de requisitos también pueden ser útiles para rastrear y evaluar cambios.
  5. Administración de Requisitos: La gestión de requisitos se ocupa del control y la evolución de los requisitos a lo largo del ciclo de vida del desarrollo.
    • Herramientas: Se utilizan sistemas de seguimiento de requisitos para gestionar cambios, trazar relaciones entre requisitos y gestionar versiones. La comunicación efectiva entre los miembros del equipo y las partes interesadas es esencial.
